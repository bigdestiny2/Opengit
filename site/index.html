What it does — proven, not aspirational

Drop-in git

git clone opengit://<key> / git push via the git-remote-opengit helper. No GitHub, no server. Proven live across two machines.

Signed issues & PRs, cross-party

Ed25519-signed, Autobase-applied, replicated between maintainer and contributor with opengit collab. Proven on the real DHT.

Fork → fetch → merge

A contributor pushes to their own opengit:// fork; the owner fetches and merges. No multi-writer needed. Dry-run-proven 11/11 with this repo.

Private repos

Per-block AEAD encryption; collaborators recover the content key over the swarm from only their identity + the repo key (cold-bootstrap).

Owner-offline availability

Blind-peer / relay pinning keeps a repo cloneable with the owner offline. Anyone runs a relay — no foundation.

Browsable

Render a repo to a zero-JS static site for PearBrowser via hyper:// — or any browser, offline.

How it works

The Corestore is the source of truth. git can't read it, so for the duration of a git operation Opengit regenerates a throwaway bare-git shadow, lets stock git upload-pack/receive-pack do the smart-protocol work, and syncs back. Discovery hangs off one plaintext manifest core whose key is the repo address.

       git  ⇄  git-remote-opengit  ⇄  ShadowRepo (bare .git on disk)
                                          ⇅   (regenerable cache)
                                     OpengitRepo ── Corestore (source of truth)
                                          ⇅                ⇅  Hyperswarm (DHT)
                              manifest · refs · objects · meta · meta-keys
                              issues-Autobase · prs-Autobase

No server. No foundation. No telemetry.

The repo is its manifest key. Naming is local-first petnames — no global namespace to capture. Relays are trusted by explicit pubkey-pinning. Apache-2.0 throughout; AGPL only via the one explicit --use-hiverelay opt-in.
